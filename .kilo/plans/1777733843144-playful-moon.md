# Plan: Switch auto_review from Claude Code to Kilo

## Goal

Migrate this repository from Claude Code project configuration to first-class Kilo Code support while improving the engineering-discipline model. Do not perform a 1:1 port of Claude hooks/plugins. Use Kilo's stronger surfaces: project plugins, agents, skills, workflows, rules, permissions, and AGENTS.md.

Priorities: correctness, cost, speed, and strict Kent-Beck-style BDD/TDD.

## Research Findings

- Current `auto_review` Claude setup is shallow: `.claude/settings.json` enables advisory pre-tool hooks plus broad Claude plugin packs; `.claude/settings.local.json` is local-only; `CLAUDE.md` is the active project guidance.
- The existing hooks are not worth preserving as-is. They only nag before edits or top-level PR comments.
- `../stochastic_macro` shows the desired spirit: small explicit rule files, outside-in TDD, RGR-shaped plans, specialist reviewers, and PR-feedback processing with mandatory reflection and guardrail-gap classification.
- Kilo supports `AGENTS.md`, `kilo.json`, `.kilo/rules/`, `.kilo/skills/`, `.kilo/agent/`, `.kilo/command/`, and `.kilo/plugin/`. Do not use `.kilocode/` or `.opencode/`.
- Kilo plugins are the best replacement for Claude hooks because they can add tools, intercept tool calls, block dangerous operations, inspect slash-command execution, inject shell env, and preserve context through compaction.
- Kilo Marketplace has useful skills/modes such as `frontend-design`, `code-simplifier`, `code-reviewer`, `test-engineer`, `changelog-generator`, and MCPs like `git`, `sqlite`, and `playwright`; none is a direct Forgejo/TDD/Rust replacement. Prefer project-local plugins and project-specific skills for the core discipline.
- Kilo hosted Code Reviews currently document GitHub/GitLab support, not Forgejo. Use Kilo local review workflows and this project's Forgejo-native tooling instead of adopting hosted Kilo PR review for this repo.

## Target Architecture

- `AGENTS.md`: concise, always-loaded project guidance replacing `CLAUDE.md`.
- `kilo.json`: registers instructions, plugins, skills, permissions, and agent defaults.
- `.kilo/rules/*.md`: short always-loaded rules only for high-frequency guardrails.
- `.kilo/skills/*/SKILL.md`: longer on-demand domain knowledge and repeatable methods.
- `.kilo/agent/*.md`: specialist primary/subagents with permissions and model/cost profiles.
- `.kilo/command/*.md`: slash-command workflows for RGR implementation, verification, review, refactor, and PR feedback.
- `.kilo/plugin/*.ts`: enforceable behavior that should not rely on model compliance alone.

## Project-Local Kilo Plugins

Implement project-local plugins rather than searching for broad third-party replacements.

1. `auto-review-discipline.ts`
- Replaces: `tdd@han`, `bdd@han`, `ralph-loop@claude-plugins-official`, `feature-dev@claude-plugins-official`, the current TDD hook, and part of `superpowers/core/bushido`.
- Adds custom tools for an RGR ledger: start cycle, record failing test output, mark green, mark refactor, inspect status.
- Intercepts edit/write/apply-patch tools. Blocks production Rust edits under `crates/*/src/*.rs` unless an active RGR cycle has recorded observed failing output for the behavior being implemented, with narrow exemptions for docs, tests, pure moves/renames, and explicitly non-behavioral chores.
- Intercepts todo/plan-style tool usage where available. Rejects component-waterfall task lists for behavior work when tasks do not name the failing test they address.
- Injects active RGR state during session compaction so a long session cannot "forget" whether RED was observed.

2. `auto-review-forgejo.ts`
- Replaces: `coderabbit@claude-plugins-official` for reply mechanics, the current review-reply hook, and GitHub-oriented assumptions from sibling patterns.
- Adds Forgejo-aware custom tools for listing reviews/comments, resolving reply targets, posting inline replies, and producing a PR-feedback status summary.
- Intercepts `bash` commands that try `gh pr comment`, `tea comment <n>`, or raw top-level PR comment endpoints. Blocks them when the operation is a reply to inline review feedback and points the agent to the inline reply tool.
- Encodes the Forgejo-specific `position`/`new_position` rule from the existing hook as enforceable behavior, not advisory prose.

3. `auto-review-toolchain.ts`
- Replaces: parts of `rust@han`, `clippy@han`, and shell/toolchain guardrails.
- Injects project-local `CARGO_HOME` and `RUSTUP_HOME` for shell commands when safe.
- Blocks `rustup` and obviously system-toolchain commands that bypass the Nix/dev-shell contract.
- Blocks unsafe git shortcuts relevant to this repo: `--no-verify`, `--no-gpg-sign`, `git add .`, `git add -A`, `git add -u`, `git commit -a`, destructive resets/checkouts, and force pushes unless explicitly approved by the user.

4. `auto-review-context.ts`
- Replaces: broad “superpowers”/memory-style plugin behavior with narrow project-specific context preservation.
- On compaction, preserves active RGR state, touched files, current verification status, unresolved Forgejo feedback, and high-risk rules currently in play.
- Optionally adds a lightweight session-finished log entry via Kilo's structured logging.

## Kilo Skills

Create project-specific skills instead of loading many always-on rules.

- `outside-in-tdd`: RGR sequence, observed-failure evidence, drill-down unit tests, and non-behavioral exemptions adapted from stochastic_macro.
- `rgr-plan-structure`: how to write implementation plans as test-addressed cycles rather than layers.
- `forgejo-feedback-protocol`: reflection, guardrail-gap vs one-off classification, inline reply mechanics, and Forgejo API details.
- `rust-workspace-engineering`: Nix toolchain, workspace crates, no unwrap/expect outside tests, error handling, env parsing, capped error bodies, and test patterns.
- `security-threat-model`: sandbox, LLM/tool trust boundaries, red-team tests, metrics/docs coupling, and when to update `docs/THREAT-MODEL.md`.
- `review-taxonomy`: adapted feedback taxonomy and cost-benefit gate so new guardrails are proportionate and do not inflate context cost.

Use marketplace skills only when clearly relevant:
- `frontend-design`: optional; only if future UI/browser-visible work appears.
- `changelog-generator`: optional helper for user/operator-facing changes.
- `agent-md-refactor` or `skill-creator`: optional one-time migration aids, not committed dependencies.

## Kilo Agents

Use fewer, sharper agents than stochastic_macro to keep costs down.

- `auto-review-rust-implementer`: primary/all agent for normal code work; strict RGR; can edit and run tests.
- `test-coverage-reviewer`: read-only subagent; checks that every new behavior was demanded by tests and that the RGR ledger/commit evidence is present.
- `security-reviewer`: read-only subagent; applies `docs/THREAT-MODEL.md`, sandbox rules, secret handling, unsafe execution, and dependency risk.
- `architecture-reviewer`: read-only subagent; checks crate boundaries, pipeline stages, public-surface docs, env-var validation, error handling, and observability contracts.
- `forgejo-feedback-processor`: subagent for PR feedback; owns reflect/classify/remediate/reply flow and uses the Forgejo plugin tools.
- `docs-operator-reviewer`: optional read-only subagent for operator-facing docs, deploy/systemd changes, and CHANGELOG consistency.

Do not add Kubernetes/Helm agents unless a concrete task touches deployment files requiring that expertise.

## Kilo Workflows

Add slash-command workflows that make the intended process easy to follow.

- `/tdd-implement`: start a behavior change, identify the smallest failing test, run it, record RED via plugin tool, implement the minimum GREEN change, run focused tests, then refactor.
- `/bugfix-rgr`: reproduce a defect with a failing test before edits, then follow RGR.
- `/refactor-safely`: require green focused tests before refactor, make one small refactor, rerun the same tests.
- `/verify`: run the repo gates through the Nix-pinned toolchain: fmt, clippy, nextest, deny, or `nix flake check` when appropriate.
- `/local-review`: dispatch architecture, test-coverage, and security reviewers over branch changes.
- `/local-review-uncommitted`: same review workflow for staged/unstaged/untracked working-tree changes.
- `/process-pr-feedback`: invoke `forgejo-feedback-processor` and require inline thread replies before any top-level summary.
- `/prepare-forgejo-pr`: scope audit, explicit-path staging guidance, CHANGELOG check, Forgejo `tea pr create` guidance, and no GitHub assumptions.

## Always-Loaded Rules

Keep always-loaded rules short and operational.

- `toolchain.md`: everything through Nix; no system rustup/cargo; project-local cargo/rustup homes.
- `tdd-discipline.md`: never write behavior production code without an observed failing test; plugin enforces edit gates.
- `forgejo.md`: Forgejo/`tea`, not GitHub/`gh`; inline review replies first.
- `scope-hygiene.md`: explicit-path staging only; no unrelated dust in scoped work.
- `security.md`: do not read secrets; sandbox and threat-model coupling.
- `verification.md`: focused test first, then broader gates before handoff.

Move long explanations into skills. This keeps default context lower than both the current Claude plugin pile and the stochastic_macro rule wall.

## Implementation Sequence

1. Remove the temporary `stochastic_macro` symlink before editing project config.
2. Create `AGENTS.md` by porting `CLAUDE.md` into Kilo-facing guidance; keep repo architecture/toolchain rules and remove Claude branding.
3. Add `kilo.json` with schema, instructions pointing at `.kilo/rules/*.md`, project plugin loading, skill paths, agent settings, and deny/ask permissions for secrets and destructive commands.
4. Add the short rules under `.kilo/rules/`.
5. Add skills under `.kilo/skills/`, using stochastic_macro content as source material but adapting to this repo's Forgejo, Nix, Rust, sandbox, and LLM architecture.
6. Add agents under `.kilo/agent/` with conservative permissions. Review agents should be read-only; implementer can edit and run focused tests.
7. Add workflows under `.kilo/command/`.
8. Develop plugins test-first where practical: extract pure helper functions for path classification, command blocking, and Forgejo reply payload construction; write small fixture tests before wiring hooks. If no JS test runner is available in the Nix shell, keep plugin logic minimal and validate with Kilo startup/config checks plus documented dry-run examples.
9. Remove `.claude/settings.json`, `.claude/settings.local.json`, and `.claude/hooks/*` after Kilo equivalents exist.
10. Search for active Claude references and migrate them. Historical references in changelog or compatibility comments can remain only if accurate; active instructions should use Kilo names.
11. Update `.gitignore` only if Kilo creates local state that should not be committed, such as `.kilo/worktrees/` or `.kilo/agent-manager.json`.

## Validation

- Run `kilo config check`.
- Run `kilo debug config` and confirm project `kilo.json`, plugins, rules, skills, agents, and commands are discovered.
- Run `kilo debug skill` and confirm project skills are present.
- Run `kilo agent list` and confirm project agents are present with expected modes/permissions.
- Start a Kilo session and confirm plugins load; inspect logs with `kilo --print-logs --log-level DEBUG` if needed.
- Dry-run discipline plugin behavior: verify production Rust edits are blocked before RED and allowed after recorded failing output.
- Dry-run Forgejo plugin behavior: verify top-level reply commands are blocked when inline feedback context exists and inline payload construction uses `position` as `new_position`.
- Use grep/search to ensure no active `.claude` references remain in project config.
- No Rust code is changed by this migration, so `nix flake check` is not required unless implementation touches Rust, Nix, CI, or generated code. If any plugin/config change affects developer commands, run the narrow relevant validation instead.

## Acceptance Criteria

- The project loads Kilo-native guidance from `AGENTS.md`, `kilo.json`, and `.kilo/`.
- Active Claude Code config is removed or inert.
- Engineering discipline is stronger than the current hooks because key behaviors are plugin-enforced, not just reminders.
- The default context is smaller than a broad plugin/rule dump: short rules are always loaded; long procedures are skills loaded on demand.
- Feature/bugfix workflows are explicitly RGR-shaped and reject waterfall task plans.
- Forgejo remains authoritative; no GitHub-only review workflow is introduced.
- Kilo plugin/agent/skill replacements for each current Claude plugin are documented and implemented in project config.
