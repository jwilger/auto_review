# Plan: Fine-Grained Outside-In RGR Workflow

## Goal
Add Kilo workflow support for the user’s preferred TDD loop: separate agents for writing RED tests, reviewing test/domain fit, clearing one diagnostic at a time, reviewing implementation quality, and returning control to the orchestrator after each micro-step.

This is a Kilo configuration/workflow change, not a Rust production behavior change.

## Existing Context
- `kilo.json` loads `AGENTS.md` and `.kilo/rules/*.md`, and registers `.kilo/skills`.
- Existing commands include `/tdd-implement`, `/bugfix-rgr`, `/verify`, `/prepare-forgejo-pr`, `/local-review`, and `/local-review-uncommitted`.
- Existing agents include `auto-review-rust-implementer`, `architecture-reviewer`, `security-reviewer`, `test-coverage-reviewer`, and `forgejo-feedback-processor`.
- `.kilo/skills/` exists but currently has no files, even though commands/agents already reference skills such as `outside-in-tdd` and `rgr-plan-structure`.

## Workflow Design
The new `/outside-in-rgr` command should orchestrate one behavior through repeated microcycles.

Cycle:

1. `rgr-test-author` writes or activates the next smallest failing test and verifies RED.
2. `rgr-test-reviewer` reviews the test before production edits.
3. If reviewer vetoes, return to `rgr-test-author` with mandatory notes.
4. `rgr-diagnostic-implementer` examines exactly one current failure diagnostic.
5. If the diagnostic identifies exactly one concrete code change, the implementer makes only that change.
6. If the diagnostic is too broad or ambiguous, the implementer writes a lower-level unit test instead and hands it back through the same test review path.
7. When the observed failure changes, stop the implementer turn and return control to the orchestrator.
8. When the current test passes, `rgr-implementation-reviewer` reviews the production diff.
9. If reviewer vetoes, return to the implementer with mandatory notes.
10. If a higher-level test still fails, continue from the diagnostic implementer step.
11. If all current cycle tests pass, run focused verification and prepare the cycle for commit or handoff.

## Agent Responsibilities

### `rgr-test-author`
Mode: subagent or primary-capable agent with edit permissions.

Responsibilities:
- Write the next smallest test for the requested behavior.
- Prefer outside-in tests first, then smaller unit tests when the workflow asks for them.
- Run the narrow focused test command.
- Capture the exact RED output.
- Confirm the failure is expected.
- Treat compiler errors as valid RED when the test intentionally proposes an API or type that does not exist yet.
- Fix only test misuse of existing code; do not fix production code.
- Produce ledger-ready output: command, observed failure, expected reason, and next reviewer handoff.

### `rgr-test-reviewer`
Mode: read-only subagent.

Responsibilities:
- Review only the test/API pressure before production edits.
- Check whether the test uses existing types correctly.
- Check whether proposed new types or APIs fit architecture, crate boundaries, and coding standards.
- Distinguish intentional API pressure from accidental misuse.
- Veto tests that are too broad, incorrectly coupled, misplaced, or architecturally inconsistent.
- Provide mandatory changes when vetoing.

### `rgr-diagnostic-implementer`
Mode: edit-capable subagent.

Responsibilities:
- Read the current single failure diagnostic and ledger.
- Make the smallest production edit that removes only the immediate diagnostic.
- Do not predict future diagnostics.
- Do not opportunistically clean up, refactor, or implement adjacent behavior.
- If the failure is ambiguous or does not identify one concrete change, write a lower-level unit test instead of production code.
- Stop when the failure changes, the test passes, or the same failure remains after a mistaken edit.
- Produce ledger-ready output identifying which of those outcomes occurred.

### `rgr-implementation-reviewer`
Mode: read-only subagent.

Responsibilities:
- Review production code after a GREEN step.
- Check minimality, architecture, error handling, security-sensitive boundaries, and style.
- Check that production behavior is demanded by an observed failing test.
- Veto overbroad implementation, speculative abstractions, missing error handling, or code inconsistent with crate patterns.
- Defer to `security-reviewer`, `architecture-reviewer`, or `test-coverage-reviewer` when the diff touches their specialized domains.

## Shared Skill
Create `.kilo/skills/outside-in-rgr-microcycle.md` as the canonical protocol. Include:

- RGR ledger template.
- Definitions of valid RED, GREEN, and REFACTOR for this workflow.
- “Single diagnostic” rule.
- Reviewer veto semantics.
- Lower-level test escape hatch for ambiguous failures.
- Control-transfer rules.
- Stop conditions and blocked states.
- Verification expectations.

Suggested ledger fields:

```text
Goal:
Current test:
Focused command:
Observed failure:

Reviewer decision:
Diagnostic under treatment:
Allowed immediate change:
Result:
Next control owner:
```

## Command Definition
Create `.kilo/command/outside-in-rgr.md` with frontmatter similar to existing commands.

Suggested behavior:

- Description: “Run a fine-grained outside-in RGR workflow with specialist agents.”
- Agent: likely `auto-review-rust-implementer` as orchestrator, unless Kilo supports command-level orchestration without a single agent.
- Body should instruct the orchestrator to use `outside-in-rgr-microcycle`, dispatch the four specialist agents at the correct phases, and keep a visible RGR ledger.
- It should explicitly prohibit skipping ahead from RED to broad implementation.
- It should require focused verification before handoff.
- It should avoid committing automatically unless the user explicitly requests commits.

## RGR/TDD For This Config Change
Because this change only adds Kilo command/agent/skill documentation and does not alter Rust behavior, production-code RED is not applicable.

Use lightweight validation instead:

1. Inspect existing command and agent frontmatter conventions before editing.
2. Add the new files with valid frontmatter.
3. Read the new files back to verify syntax, permissions, and wording.
4. Optionally run any existing local Kilo/plugin validation if present in `.kilo/package.json`.
5. Do not run Rust gates unless implementation unexpectedly touches Rust or repo behavior outside `.kilo/`.

## Open Design Decisions
No blocking questions. The plan assumes:

- New specialist agents should be project-local under `.kilo/agent/`.
- The new workflow should be command-driven under `.kilo/command/`, not encoded into one mega-agent.
- Existing `auto-review-rust-implementer` remains the default agent and can serve as the command orchestrator.
- The workflow should not auto-commit unless the user explicitly asks for commit behavior in a later issue/PR workflow.

## Success Criteria
- `/outside-in-rgr` exists and clearly encodes the user’s micro-TDD workflow.
- Each specialist role has a focused, enforceable agent definition.
- Reviewers have explicit veto power.
- The diagnostic implementer is constrained to one current diagnostic and has clear stop conditions.
- The lower-level unit test escape hatch is documented.
- Existing `/tdd-implement` remains usable and points to the new stricter workflow.
